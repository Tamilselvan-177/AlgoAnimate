Flood Fill

The paint-bucket algorithm. Starting from pixel (sr, sc), repaint all 4-directionally connected pixels sharing the same original color with a new color.

⏱ ~10 min to understand

Problem Statement

You are given an image as an m × n grid of integers, and three integers sr, sc, and color.

Perform a flood fill from pixel (sr, sc):
  • Change (sr, sc) to color
  • For each 4-directional neighbour with the same original color — change it too
  • Repeat recursively for all newly changed pixels
  • Stop when no more matching neighbours exist
Return the modified image.
Example 1
image = [[1,1,1],[1,1,0],[1,0,1]], sr=1, sc=1, color=2
Output: [[2,2,2],[2,2,0],[2,0,1]]
Bottom-right 1 not connected — stays unchanged
Example 2 — edge case
image = [[0,0,0],[0,0,0]], sr=0, sc=0, color=0
Output: [[0,0,0],[0,0,0]]
origColor == newColor → return immediately

Constraints

1 ≤ m, n ≤ 50
0 ≤ image[i][j], color < 2¹⁶
0 ≤ sr < m, 0 ≤ sc < n
Complexity
Time: O(m × n)
Space: O(m × n)
Every cell visited at most once

Key observations

If origColor == color, return immediately — else BFS loops forever.

Paint on enqueue not on dequeue — prevents re-enqueueing the same cell.

4-directional only — diagonal neighbours do not count.

from collections import deque

def floodFill(image, sr, sc, color):
    origColor = image[sr][sc]

    # Edge case: already the target color
    if origColor == color:
        return image

    rows, cols = len(image), len(image[0])
    q = deque([(sr, sc)])
    image[sr][sc] = color          # paint on enqueue

    dirs = [(1,0), (-1,0), (0,1), (0,-1)]

    while q:
        r, c = q.popleft()
        for dr, dc in dirs:
            nr, nc = r + dr, c + dc
            if (0 <= nr < rows and
                0 <= nc < cols and
                image[nr][nc] == origColor):
                image[nr][nc] = color  # paint on enqueue
                q.append((nr, nc))

    return image

def floodFill(image, sr, sc, color):
    origColor = image[sr][sc]
    if origColor == color: return image

    def dfs(r, c):
        if (r < 0 or r >= len(image) or
            c < 0 or c >= len(image[0]) or
            image[r][c] != origColor):
            return
        image[r][c] = color
        dfs(r+1, c); dfs(r-1, c)
        dfs(r, c+1); dfs(r, c-1)

    dfs(sr, sc)
    return image
💡 Flood Fill = BFS with a color-match condition. The grid mutation doubles as the visited marker — no extra set needed.

Step 1 — The paint bucket analogy

Exactly like the bucket fill tool in MS Paint. Click a pixel → every touching pixel of the same color gets repainted. Stops at color boundaries.
What spreads
From (sr, sc), any pixel with image[r][c] == origColor that is 4-directionally reachable gets repainted. Diagonal pixels are ignored.
What stops it
Pixels with a different color act as walls. BFS never crosses them. Disconnected regions of the same color are also not touched.

Step 2 — Paint on enqueue, not on dequeue

This is the most important implementation detail. We recolor the cell immediately when we add it to the queue, not when we process it.
Wrong (paint on dequeue): same cell can be added to queue multiple times from different neighbours → wasteful or infinite loop
Correct (paint on enqueue): cell color changes immediately → next neighbour check sees new color → never re-enqueued
The grid is the visited set. Recoloring replaces visited.add(r,c) — same idea used in Rotting Oranges.

Step 3 — The critical edge case: origColor == color

If the start pixel already has the target color, return immediately. Without this check:
1. Enqueue (sr, sc), paint it to color (= origColor, no change)
2. Pop (sr, sc), check neighbours — all match origColor (= color)
3. Enqueue all neighbours, paint them (still same color)
4. Infinite loop — the grid never stops matching
Fix: if origColor == color: return image — one line, saves everything

Step 4 — Why 4-directional, not 8?

The problem specifies 4-directional adjacency (up, down, left, right). Diagonal pixels are not connected.
Example 1 — why bottom-right 1 stays unchanged
1 1 1
1 1 0
1 0 1
The amber 1 at (2,2) is only diagonally adjacent to the flood region.
Its only 4-directional neighbours are (1,2)=0 and (2,1)=0 — both walls.
Never reached → stays unchanged.

Step 5 — BFS vs DFS: does it matter?

BFS (queue)
Spreads level by level outward. Iterative — no stack overflow risk. O(m×n) queue space.
DFS (recursion)
Dives deep first. Elegant and short. Risk: stack overflow on large 50×50 grids (2500 deep).
Both give the same final result — the entire connected region is repainted. BFS is safer for large grids; DFS is more intuitive for interviews.
The pattern: Flood Fill, Rotting Oranges, Number of Islands, and 01 Matrix all share the same skeleton — BFS on a grid with a match condition, recolor/mark on enqueue. Master the pattern once, apply it everywhere.
