Diagonal Traverse

Return every cell of an m × n matrix in diagonal zig-zag order: walk ↗ and ↙, bouncing off the borders. The classic simulation keeps O(m·n) time and O(1) extra space (besides the output).

⏱ ~10 min to understand

Problem Statement

Given an m × n integer matrix mat, return a 1D array listing all elements in diagonal order (LeetCode 498): start at (0,0), repeatedly move along diagonals, alternating direction so the path zig-zags across the grid.
Example 1 — 3 × 3 (classic)
Input: mat = [[1,2,3],[4,5,6],[7,8,9]]
Output: [1,2,4,7,5,3,6,8,9]
Anti-diagonals (same i+j): [1] → [2,4] → [7,5,3] → [6,8] → [9]. Stripes alternate: e.g. k=1 reads top→bottom (2,4); k=2 reads bottom→top (7,5,3).
Example 2 — 2 × 2 (single diagonal stripes)
Input: mat = [[1,2],[3,4]]
Output: [1,2,3,4]
Order: 1 → (2,3) on one diagonal → 4. No long “middle” diagonal — still the same edge rules.

Constraints

m == mat.length, n == mat[i].length
1 ≤ m, n ≤ 10⁴ and 1 ≤ m·n ≤ 10⁴
−10⁵ ≤ mat[i][j] ≤ 10⁵
Complexity
Time: O(m·n) — visit each cell once
Space: O(1) extra (output array O(m·n) is required)
Key insight
A boolean up tells you whether the next move is ↗ or ↙. When the next step would leave the grid, slide along the top / bottom / left / right edge instead and flip up. Order of if checks matters at corners.
Edge simulation
By diagonal sum (i + j)
Simulate position (r, c) and direction
O(m·n)O(1)
def findDiagonalOrder(mat):
    if not mat: return []
    m, n = len(mat), len(mat[0])
    res = []
    r, c = 0, 0
    up = True
    while len(res) < m * n:
        res.append(mat[r][c])
        if up:
            if c == n - 1: r += 1; up = False
            elif r == 0: c += 1; up = False
            else: r -= 1; c += 1
        else:
            if r == m - 1: c += 1; up = True
            elif c == 0: r += 1; up = True
            else: r += 1; c -= 1
    return res
Why this matches the statement
While moving ↗, hitting the right border forces a step down; hitting the top (and not the right corner) forces a step right — then toggle to ↙. The ↙ branch mirrors that for bottom vs left. This is exactly what the animation steps through.
Bucket by anti-diagonal, reverse every other bucket
O(m·n)O(m·n)
from collections import defaultdict

def findDiagonalOrder(mat):
    m, n = len(mat), len(mat[0])
    d = defaultdict(list)
    for i in range(m):
        for j in range(n):
            d[i + j].append(mat[i][j])
    res = []
    for k in range(m + n - 1):
        if k % 2 == 0:
            res.extend(reversed(d[k]))
        else:
            res.extend(d[k])
    return res
All cells with the same i + j sit on one anti-diagonal. For even k, LeetCode’s order reads that diagonal from bottom to top (reversed); for odd k, top to bottom. Extra memory for the buckets, but very clear in interviews.

Step 1 — Anti-diagonals share i + j

Index k = i + j is constant along a line from upper-right to lower-left. Example 1’s matrix has diagonals k = 0…4. The problem’s zig-zag is equivalent to visiting each group of k in order, alternating whether you list cells with increasing or decreasing row.

Step 2 — Simulation: one step at a time

After recording mat[r][c], decide the next cell using up. If up is true, you prefer (r-1, c+1); if that would leave the grid, apply the edge rule (right column → go down; top row → go right), then set up = False. The else branch handles ↙ symmetrically. Corners are covered because you test the “blocking” edge first (e.g. right column before top row when moving up).

Step 4 — Pitfalls

✗ Wrong if order at corners
When up is true, check c == n-1 before r == 0 so the top-right corner steps down, not right off the grid.
✗ Off-by-one in bucket method
Diagonals are k = 0 … m+n-2. Reversing the wrong parity of k mirrors the traversal — compare with a 2×2 example.
