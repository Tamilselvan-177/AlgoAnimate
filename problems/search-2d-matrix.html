Search a 2D Matrix

Each row is sorted, and each row's first element is greater than the last of the previous row. That means the whole matrix is one sorted flat array in disguise — binary search it directly in O(log m·n).

⏱ ~8 min to understand

Problem Statement

Given an m × n integer matrix with two properties:

  • Each row is sorted in non-decreasing order
  • The first integer of each row is greater than the last integer of the previous row

Given an integer target, return true if target is in the matrix, false otherwise. Must run in O(log(m·n)).
Example 1
matrix = [[1,3,5,7],[10,11,16,20],[23,30,34,60]]
target = 3 → true
Example 2
target = 13 → false

Constraints

m == matrix.length
n == matrix[i].length
1 ≤ m, n ≤ 100
−10⁴ ≤ matrix[i][j], target ≤ 10⁴
Must be O(log(m·n))
Key insight
The matrix is a sorted 1D array folded into 2D. Index k maps to row=k÷n, col=k%n. One binary search covers everything.
Flat binary search
Two-pass (row then col)
Treat as flat sorted array — one binary search
O(log m·n)O(1)
from typing import List

def searchMatrix(matrix: List[List[int]], target: int) -> bool:
    if not matrix or not matrix[0]: return False  # edge case
    m, n = len(matrix), len(matrix[0])
    lo, hi = 0, m * n - 1

    while lo <= hi:
        mid = lo + (hi - lo) // 2  # overflow-safe
        val = matrix[mid // n][mid % n]

        if   val == target: return True
        elif val  < target: lo = mid + 1
        else:               hi = mid - 1

    return False
Index mapping
flat index k → row = k // n, col = k % n
k=5, n=4 → row=1, col=1 ✓
Why overflow-safe?
Python ints don't overflow, but lo+(hi-lo)//2 is the correct habit for Java/C++ where (lo+hi)//2 can overflow 32-bit int.
Two-pass: binary search row, then column
O(log m + log n)O(1)
from typing import List

def searchMatrix(matrix: List[List[int]], target: int) -> bool:
    if not matrix or not matrix[0]: return False
    m, n = len(matrix), len(matrix[0])

    # Step 1: find the row target could be in
    top, bot = 0, m - 1
    while top <= bot:
        row = top + (bot - top) // 2
        if   matrix[row][-1] < target: top = row + 1
        elif matrix[row][0]  > target: bot = row - 1
        else: break
    else: return False   # no valid row found

    # Step 2: binary search within that row
    lo, hi = 0, n - 1
    while lo <= hi:
        mid = lo + (hi - lo) // 2
        if   matrix[row][mid] == target: return True
        elif matrix[row][mid]  < target: lo = mid + 1
        else:                            hi = mid - 1
    return False
O(log m + log n) = O(log m·n) — same asymptotic complexity as flat approach, two calls instead of one.

Step 1 — The key observation: it's a 1D array

Because each row is sorted AND each row starts after the previous ends, the entire matrix is equivalent to one sorted array of length m·n — just folded into rows.

k=0..11   fully sorted ✓

Step 2 — Index mapping: flat ↔ 2D

Given flat index k in an m×n matrix:
row = k // n   # integer division
col = k % n   # remainder
k=5, n=4 → row=1, col=1 → matrix[1][1]=11 ✓

Step 3 — Standard binary search on flat indices

lo=0, hi=m·n−1. At each step compute mid, map to 2D, compare with target.
val = matrix[mid // n][mid % n]
val == target → return True
val < target → lo = mid + 1  (search right)
val > target → hi = mid − 1  (search left)
Why O(log m·n) = O(log m + log n)
log(m·n) = log m + log n  — logarithm product rule. A 100×100 matrix needs at most log₂(10000) ≈ 14 comparisons, not 10000.

Step 4 — Full trace: target = 3

Matrix 3×4, flat length=12, target=3
lo=0, hi=11 → mid=5 → matrix[1][1]=11 > 3 → hi=4
lo=0, hi=4 → mid=2 → matrix[0][2]=5 > 3 → hi=1
lo=0, hi=1 → mid=0 → matrix[0][0]=1 < 3 → lo=1
lo=1, hi=1 → mid=1 → matrix[0][1]=3 == 3 → return True ✓
4 comparisons for 12 elements — O(log 12) ≈ 4 ✓

Step 5 — Common mistakes

✗ Empty matrix crash
matrix[0] raises IndexError if matrix=[] or matrix=[[]]. Always guard: if not matrix or not matrix[0]: return False
✗ Wrong index formula
mid // m instead of mid // n — divide by number of columns (n), not rows (m). Confusing m and n is the #1 bug here.
✗ Using this for LC 240
LC 240 rows are sorted but rows don't chain — the flat binary search doesn't apply. Use staircase search from top-right corner instead.
✗ Overflow-unsafe mid (C++/Java)
In languages with 32-bit int, (lo+hi)//2 overflows when both exceed 2³¹. Always write lo + (hi-lo)//2.
The flat-array insight: Whenever a 2D structure has total ordering (each row/col chains into the next), treat it as 1D. The O(log m·n) = O(log m + log n) — same as two separate binary searches, but in one pass.
