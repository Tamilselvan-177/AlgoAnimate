Implement Queue
using Stacks

Simulate a FIFO queue using only two LIFO stacks. The trick: one stack handles push, the other handles pop/peek. Elements flow from Inbox → Outbox when the outbox is empty — making each operation amortized O(1).

⏱ ~12 min to understand

Problem Statement

Implement a FIFO queue using only two stacks. The queue must support:

push(x) — adds x to the back of the queue.
pop() — removes and returns the front element.
peek() — returns the front element without removing it.
empty() — returns true if the queue is empty.

You may only use standard stack operations: push to top, peek/pop from top, size, and is empty.
Example 1
Input: push(1) → push(2) → peek() → pop() → empty()
Output: [null, null, 1, 1, false]
After push(1), push(2): queue = [1, 2]. peek() returns 1 (front). pop() returns 1, queue = [2]. empty() → false.

For the Inbox/Outbox picture, stack vs queue, and why operations are amortized O(1), see the Explanation tab.

class MyQueue:
    def __init__(self):
        self.s1 = []  # inbox  — new elements go here
        self.s2 = []  # outbox — elements served from here

    def push(self, x: int) -> None:
        self.s1.append(x)   # always push to inbox

    def _transfer(self):
        # move ALL of inbox to outbox (reverses order)
        if not self.s2:
            while self.s1:
                self.s2.append(self.s1.pop())

    def pop(self) -> int:
        self._transfer()
        return self.s2.pop()

    def peek(self) -> int:
        self._transfer()
        return self.s2[-1]

    def empty(self) -> bool:
        return not self.s1 and not self.s2

class MyQueue:
    def __init__(self):
        self.s1 = []  # always holds correct order
        self.s2 = []  # temp helper during push

    def push(self, x: int) -> None:
        # move everything out, push new, put back
        while self.s1:
            self.s2.append(self.s1.pop())
        self.s1.append(x)
        while self.s2:
            self.s1.append(self.s2.pop())

    def pop(self) -> int:
        return self.s1.pop()

    def peek(self) -> int:
        return self.s1[-1]

    def empty(self) -> bool:
        return not self.s1

Stack vs Queue

Stack (LIFO)
Last in → First out
push/pop same end
Queue (FIFO)
First in → First out
push back, pop front
📊 How two stacks simulate a queue
Key insight: When you push 1, 2, 3 into Inbox (s1), the order is reversed (3 on top). Pouring everything into Outbox (s2) reverses it again — putting 1 on top, which is the correct front of the queue. This double-reversal is the entire trick.
Amortized O(1) — intuition
Amortized O(1): each element is moved from s1→s2 exactly once across its entire lifetime. n pushes + n pops = 2n moves total = O(1) per op on average.

Step 1 — The double-reversal trick

A stack reverses order. Push 1, 2, 3 → they come out 3, 2, 1 (last in, first out).

But if you do it twice, you get the original order back: pour stack-A into stack-B, then pop from B — you get 1 first, then 2, then 3. That's exactly what a queue needs.
Push 1,2,3
s1 top→bot
3, 2, 1
⟹
Pour s1→s2
s2 top→bot
1, 2, 3
⟹
Pop from s2
First out: 1
✓ FIFO!

Step 2 — Lazy vs Eager transfer

✓ Lazy (preferred)
Transfer only when s2 is empty.
push → always O(1)
pop/peek → O(n) worst, amortized O(1)
Each element moves s1→s2 exactly once.
Eager
Transfer on every push.
push → O(n) always
pop/peek → O(1) always
More expensive overall.

The Animation tab implements the lazy design only (inbox/outbox pour on pop/peek when s2 is empty). The eager tab is a different algorithm — same FIFO idea, different cost profile.

Step 3 — Amortized O(1) proof

Consider n operations. Each element is pushed onto s1 once (n pushes). Each element is moved from s1 to s2 at most once (n moves). Each element is popped from s2 at most once (n pops).
Total work for n elements
n pushes to s1    = O(n)
n transfers s1→s2 = O(n)
n pops from s2   = O(n)
Total = 3n = O(n)  →  per operation = O(1) amortized

peek / pop vs empty stacks (three cases)

  • s2 not empty — No pour. FRONT is already s2 TOP; pop/peek uses s2 only.
  • s2 empty, s1 not empty — Pour all s1→s2 once, then pop/peek from s2. This is the “double reversal” moment.
  • s2 empty, s1 empty — Queue is empty. (LeetCode guarantees you won’t pop/peek in this state.)

Pattern: "Inbox / Outbox" is the mental model. New mail goes in Inbox. When you need to deliver, pour Inbox into Outbox (oldest mail now on top). The Outbox drains completely before you refill it — this is what guarantees FIFO order and amortized O(1).
